# 1cedo — Конвертер документов ЭДО 1С → Диадок

Веб-сервис на Django (без БД) для очистки XML-документов ЭДО, выгруженных из 1С, от служебных полей перед отправкой в Диадок.

## Задача

При выгрузке из 1С в XML-документы добавляются поля `ИнфПолФХЖ1` и `ИнфПолФХЖ2` с внутренними идентификаторами 1С. Диадок их не принимает. Сервис получает zip-архив от пользователя, находит в нём XML-файл основного документа (имя начинается с `ON_`), удаляет указанные поля и отдаёт очищенный XML.

## Стек

- Python 3.10+
- Django 4.2 (без БД — `django.db.backends.dummy`)
- lxml (единственная внешняя зависимость помимо Django)

## Архитектура

### Почему без БД
Сервис не хранит состояние. Все операции выполняются в памяти по принципу
«загрузил → обработал → скачал». Используется `django.db.backends.dummy`.

### Обработка zip
- Файл читается в память через `BytesIO`, на диск не пишется.
- В архиве ищется ровно один XML-файл с именем, начинающимся на `ON_`.
  Остальные файлы (`meta.xml`, `card.xml`) игнорируются.

### Обработка XML
- Парсинг через `lxml.etree` (поддерживает `windows-1251`).
- Удаление элементов через `etree.strip_elements(root, 'ИмяЭлемента')` —
  удаляет сам элемент и всех его потомков.
- Сериализация обратно с сохранением оригинальной кодировки и XML-декларации.
- Пространства имён (`xmlns`) у удаляемых элементов отсутствуют, что упрощает поиск.

### Имя выходного файла

Если в XML найдены оба атрибута:
- `НаимДокОпр` у элемента `Документ`,
- `НомерДок` у элемента `СвСчФакт`,

то выходной файл именуется `<НаимДокОпр>_<НомерДок>.xml`, где пробелы и `/` заменены на `_`. Кавычки в значениях не заменяются.

Если хотя бы один атрибут отсутствует — используется оригинальное имя XML-файла из архива.

### Обработка ошибок

| Ситуация | Поведение |
|----------|-----------|
| Загружен не zip | Ошибка валидации формы |
| Размер файла > 64 Кбайт | Ошибка валидации формы |
| В архиве нет файла `ON_*` | Сообщение об ошибке на странице |
| XML повреждён/невалиден | Сообщение об ошибке на странице |

## Запуск

Системные пакеты Debian Trixie: `python3-django`, `python3-lxml`.
Никаких `venv`, `pip` и `requirements.txt` — всё уже есть в системе.

```bash
# Установка (если ещё нет)
sudo apt install python3-django python3-lxml

# Без миграций (БД не нужна)
python3 manage.py runserver
```

## Тестирование

```bash
python manage.py test
```

Тесты в `converter/tests.py` используют `django.test.SimpleTestCase`.
Тестовые данные — архив `data/1111.zip`.

### Набор тестов

| Тест | Что проверяет |
|------|---------------|
| `test_valid_zip_removes_infpol` | Загружаем валидный zip → ответ 200, в XML нет `<ИнфПолФХЖ1>` и `<ИнфПолФХЖ2>`, кодировка `windows-1251` |
| `test_non_zip_upload_rejected` | Загружаем файл без расширения `.zip` → ошибка валидации формы |
| `test_zip_without_on_file` | Загружаем zip, в котором нет файла с префиксом `ON_` → ошибка обработки |
| `test_download_filename` | Имя скачиваемого файла соответствует имени исходного XML |

Сами тесты лежат рядом с реальными файлами — данные из `data/1111.zip`.

## Кодировка

Все XML-файлы из 1С используют кодировку `windows-1251` (cp1251).
При чтении и записи всегда явно указывается кодировка.
Выходной XML содержит `<?xml version="1.0" encoding="windows-1251"?>`.

## Решения, принятые осознанно

1. **Django, а не Flask** — требование задачи.
2. **lxml вместо xml.etree.ElementTree** — lxml корректно сохраняет XML-декларацию с кодировкой windows-1251 при сериализации; стандартный ElementTree в некоторых версиях Python отбрасывает её.
3. **strip_elements вместо итеративного remove** — один вызов, удаляет все вхождения элемента по имени, включая вложенные (теоретически возможный случай).
4. **Вся обработка в памяти** — нет смысла писать на диск для одноразовой операции.
